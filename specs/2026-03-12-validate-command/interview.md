## Context

The FineType CLI has three user-facing workflows: `profile` (discovery), `load` (analytics/DuckDB CTAS), and `schema` (type-level JSON Schema export). Validation is currently split awkwardly â€” `profile --validate` runs inline quality checks, and a separate `validate` subcommand does per-value NDJSON repair with quarantine/null/ffill/bfill strategies.

Hugh wants to separate validation into a proper schema-driven quality gate, using JSON Schema as the reusable artefact that bridges one-off profiling and repeatable pipeline validation. The schema should be auto-generated by FineType (not hand-crafted) and editable by users.

Key references: qsv's `schema` â†’ `validate` workflow ([schema docs](https://github.com/dathere/qsv/blob/master/docs/help/schema.md), [validate docs](https://github.com/dathere/qsv/blob/master/docs/Validate.md)), DuckDB's [rejects tables](https://duckdb.org/docs/stable/data/csv/reading_faulty_csv_files), and Hugh's own [qsv #1728](https://github.com/dathere/qsv/issues/1728) on predictable output in the all-valid case.

## Interview Q&A

### Q1: Use case
**Q:** When you say 'reproducible local analytics', what's the user story?
**A:** Both â€” schema as a reusable artefact. Profile once â†’ save schema â†’ validate future batches. Also schema feeding into local pipeline steps (profile â†’ validate â†’ load).

### Q2: Schema source
**Q:** Today's `finetype schema` exports JSON Schema per-type. A validate command needs a table-level schema â€” column names mapped to types. Where should that come from?
**A:** It's still JSON Schema, just at the table level. Referenced qsv's `schema` command which auto-generates table-level JSON Schema from CSV data via `stats`. FineType should follow the same pattern â€” auto-generate, user can edit.

### Q3: Schema depth
**Q:** What level of schema fidelity should FineType produce â€” type-level (taxonomy patterns) or data-level (observed min/max/cardinality/nullability)?
**A:** Type-level default, data-level opt-in. `schema data.csv` emits type-level constraints from taxonomy definitions. `schema data.csv --stats` adds observed data constraints.

### Q4: Output model
**Q:** Should validate produce sidecar files (qsv-style), structured report to stdout, or both?
**A:** Referenced qsv's sidecar convention (`.valid`, `.invalid`, `.validation-errors.jsonl`) and DuckDB's rejects tables. Both patterns capture error messages per row, which is important. Need to handle the all-valid case carefully.

### Q5: All-valid case
**Q:** What should happen when all rows are valid?
**A:** Referenced [qsv #1728](https://github.com/dathere/qsv/issues/1728) â€” an issue Hugh filed. qsv's original behavior didn't produce a `.valid` file when all rows passed, breaking CI/CD pipelines. The fix was `--valid-output`. Lesson: validate must always produce predictable output files regardless of result.

### Q6: Output contract
**Q:** Stdout default or sidecar files always?
**A:** Sidecar files always. Always write `.valid.csv` + `.invalid.csv` + `.errors.jsonl`. Stdout gets the summary report. Predictable for automation.

### Q7: Existing validate command
**Q:** What happens to the existing `finetype validate` with its quarantine/repair strategies?
**A:** Replace it entirely. The quarantine/null/ffill/bfill repair strategies were an experiment. The new schema-driven validate is a quality gate (pass/fail), not a repair tool.

### Q8: CLI surface â€” where does `finetype schema` fit?
**Q:** The existing `finetype schema` exports type-level JSON Schema. The proposed table-level schema is a different artefact. Where should table-level schema generation live?
**A:** Extend `schema` to do both. `finetype schema identity.person.email` = type-level (as today). `finetype schema data.csv` = table-level (detects file input, runs profile internally). One command, context-dependent.

### Q9: Full CLI surface
**Q:** Does this capture the right flow?
```
finetype schema identity.person.email    # Type-level JSON Schema (as today)
finetype schema data.csv                 # Table-level JSON Schema (profile + emit)
finetype schema data.csv --stats         # Table-level + observed data constraints
finetype validate data.csv schema.json   # Quality gate â†’ .valid/.invalid/.errors.jsonl
finetype profile data.csv                # Discovery (no schema output)
finetype load data.csv                   # Analytics / DuckDB CTAS
```
**A:** Yes, and remove `--validate` from the `profile` command. Also preferred `--stats` over `--strict` for the data-constraint flag.

### Q10: Flag naming
**Q:** What name for the observed-constraints flag?
**A:** `--stats`. Short, familiar, qsv ecosystem has primed users to expect stats-derived schema enrichment.

### Q11: Scope
**Q:** What does 'done' look like?
**A:** Full pipeline â€” schema + validate MVP, stats enrichment, plus validate integration in MCP server tools and DuckDB extension.

### Q12: Sidecar file opt-out
**Q:** Sidecar files always could lead to undesirable outcomes when it can't be turned off. Should there be a flag to suppress file creation while still showing a summary?
**A:** Yes. `--summary-only` suppresses sidecar file creation. Stdout always shows the summary report regardless of flag. Self-documenting name preferred over `--quiet` or `--no-output`.

## Summary

### Goal
Separate validation from profiling into a standalone, schema-driven quality gate. Extend `finetype schema` to auto-generate table-level JSON Schema from CSV data. New `finetype validate` consumes that schema to produce sidecar files (`.valid.csv`, `.invalid.csv`, `.errors.jsonl`) with predictable output in all cases.

### Constraints
- JSON Schema Draft 2020-12 as the schema standard (matches qsv ecosystem)
- Always produce sidecar files â€” no conditional file creation (lesson from qsv #1728)
- Error detail as JSONL (one object per line, streamable, like qsv + DuckDB rejects)
- `finetype schema` detects input type: type key â†’ type-level, file path â†’ table-level
- Replace the existing `validate` command entirely (no quarantine/repair strategies)
- Remove `--validate` flag from `profile` command

### Success Criteria
- `finetype schema data.csv` emits valid JSON Schema with FineType type constraints per column
- `finetype schema data.csv --stats` adds observed min/max/cardinality/null-rate constraints
- `finetype validate data.csv schema.json` produces `.valid.csv`, `.invalid.csv`, `.errors.jsonl`
- `--summary-only` suppresses sidecar files; stdout summary always prints
- All-valid case: `.valid.csv` = full input, `.invalid.csv` = headers only, `.errors.jsonl` = empty
- Exit code: 0 = all valid, 1 = has errors
- Summary report to stdout (row counts, per-column pass rates)
- MCP server `validate` tool exposes the same capability
- DuckDB extension supports schema-driven validation
- Old `validate` command and `profile --validate` flag removed

### Open Questions
- JSON Schema `$id` / `$schema` conventions for generated table schemas
- Whether `finetype schema` should write to sidecar file (`.schema.json`) or stdout by default
- Error object shape â€” per-row (qsv-style) or per-cell (DuckDB rejects-style)?
- How `validate` handles columns present in data but absent from schema (and vice versa)
- Whether `load` should optionally consume a schema (skip re-profiling)
